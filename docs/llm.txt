# Agent Orcha - LLM Documentation

## Project Overview

Agent Orcha is a declarative TypeScript framework for building, managing, and scaling multi-agent AI systems. It combines the flexibility of TypeScript with the simplicity of YAML to orchestrate complex workflows, manage diverse tools via Model Context Protocol (MCP), and integrate semantic search and knowledge graphs seamlessly.

**Repository:** https://github.com/ddalcu/agent-orcha
**NPM Package:** https://www.npmjs.com/package/agent-orcha
**Version:** 0.0.8
**License:** MIT
**Requirements:** Node.js >= 24.0.0

## Key Features

1. **Declarative AI**: Define agents, workflows, and infrastructure in version-controlled YAML files
2. **Model Agnostic**: Swap between OpenAI, Gemini, Anthropic, or local LLMs (Ollama, LM Studio) without code changes
3. **Universal Tooling**: Leverage Model Context Protocol (MCP) to connect agents to any external service, API, or database
4. **Knowledge Stores**: Unified SQLite + sqlite-vec persistence layer with optional direct graph mapping for semantic search and entity analysis
5. **ReAct Workflows**: Autonomous prompt-driven workflows with ReAct/single-turn modes, tool/agent discovery, and human-in-the-loop
6. **Robust Workflow Engine**: Orchestrate complex multi-agent sequences with parallel execution, conditional logic, and state management
7. **Agent Orcha Studio**: Built-in web dashboard with agent testing, knowledge browsing, workflow execution, MCP management, and in-browser IDE
8. **Conversation Memory**: Session-based memory with FIFO management and automatic TTL cleanup
9. **Structured Output**: JSON schema validation for agent responses
10. **Extensible Functions**: Custom JavaScript functions to extend agent capabilities with zero boilerplate
11. **AI-First Design**: YAML configurations are ideal for LLMs to read, write, and maintain
12. **Published Agents**: Share agents via standalone chat pages at /chat/<name> with optional per-agent password protection
13. **Browser Sandbox**: Full Chromium browser with CDP control, Xvfb, and noVNC — plus experimental Vision Browser for pixel-coordinate control
14. **Skills System**: Prompt augmentation via Markdown files injected into agent system prompts
15. **Integrations**: Connect agents to external platforms (Email via IMAP/SMTP, Collabnook WebSocket chat)
16. **Triggers**: Cron schedules and webhooks for automated agent execution
17. **Engine Registry**: Pluggable local inference with llama-cpp, mlx-serve, Ollama, and LM Studio
18. **Environment Variable Substitution**: ${ENV_VAR} and ${ENV_VAR:-default} in all config files
19. **Thinking/Reasoning**: Extended thinking for Anthropic, reasoning budgets for local models
20. **Security**: Rate limiting, SSRF protection, SQL injection hardening

## Quick Start

### 1. Initialize Project

```bash
npx agent-orcha init my-project
cd my-project
```

This creates a project with:
- agents/ - Agent configuration files
- workflows/ - Workflow definition files
- functions/ - Custom JavaScript functions
- knowledge/ - Knowledge store configurations and data
- skills/ - Skill prompt augmentation files
- llm.json - LLM and embedding provider settings
- mcp.json - MCP server configuration

### 2. Configure LLM (llm.json)

```json
{
  "version": "1.0",
  "models": {
    "default": "llama-cpp",
    "llama-cpp": {
      "provider": "local",
      "engine": "llama-cpp",
      "model": "gemma-4-E2B-it-IQ4_NL",
      "reasoningBudget": 0,
      "contextSize": 32768
    },
    "ollama": {
      "provider": "local",
      "engine": "ollama",
      "baseUrl": "http://localhost:11434/v1",
      "model": "qwen3.5:latest",
      "reasoningBudget": 0
    },
    "openai": {
      "provider": "openai",
      "apiKey": "${OPENAI_API_KEY}",
      "model": "gpt-4o"
    },
    "anthropic": {
      "provider": "anthropic",
      "apiKey": "${ANTHROPIC_API_KEY}",
      "model": "claude-sonnet-4-6",
      "thinkingBudget": 10000
    }
  },
  "embeddings": {
    "default": "llama-cpp",
    "llama-cpp": {
      "provider": "local",
      "engine": "llama-cpp",
      "model": "nomic-embed-text-v1.5.Q4_K_M",
      "eosToken": "</s>"
    },
    "openai": {
      "provider": "openai",
      "apiKey": "${OPENAI_API_KEY}",
      "model": "text-embedding-3-small"
    }
  },
  "engineUrls": {
    "lmstudio": "http://localhost:1234",
    "ollama": "http://localhost:11434"
  }
}
```

**Pointer System:** The `default` key is a string reference to another entry (e.g., `"default": "llama-cpp"`). This enables fast switching between pre-configured providers without duplicating config. Getter functions automatically dereference one level of pointer.

**Model Config Fields:**
- `provider` — `openai` | `anthropic` | `gemini` | `local`
- `engine` — `llama-cpp` | `mlx-serve` | `ollama` | `lmstudio` (for `provider: local`)
- `model` — Model name or filename
- `baseUrl` — Custom API endpoint (optional for cloud providers)
- `apiKey` — API key (supports `${ENV_VAR}` substitution)
- `temperature` — Sampling temperature (optional)
- `maxTokens` — Max generation tokens (optional, default 4096 for local)
- `contextSize` — Context window size (optional, auto-calculated for local)
- `reasoningBudget` — Token budget for chain-of-thought reasoning in local models (0 = disabled)
- `thinkingBudget` — Token budget for extended thinking in Anthropic models

**Embedding Config Fields:**
- `provider`, `engine`, `model`, `baseUrl`, `apiKey` — Same as model config
- `eosToken` — End-of-sequence token override (e.g., `</s>` for nomic models)

**engineUrls:** Custom base URLs for external engines (Ollama, LM Studio). Stored in `llm.json` and configurable via API.

**LLM Providers:**
- OpenAI: Omit baseUrl (uses default OpenAI endpoint)
- LM Studio: baseUrl configured via engineUrls
- Ollama: baseUrl configured via engineUrls
- Anthropic: Supported via provider configuration
- Gemini: Supported via provider configuration
- Local (llama-cpp, mlx-serve): Managed engines with automatic binary management

### 3. Create Agent (agents/myagent.agent.yaml)

```yaml
name: myagent
description: My first AI agent
version: "1.0.0"

llm:
  name: default
  temperature: 0.7

prompt:
  system: |
    You are a helpful assistant.
    Answer questions clearly and concisely.
  inputVariables:
    - query

output:
  format: text
```

### 4. Start Server

```bash
npx agent-orcha start
# Server runs on http://localhost:3000
# Agent Orcha Studio available at http://localhost:3000
```

### 5. Invoke Agent

```bash
curl -X POST http://localhost:3000/api/agents/myagent/invoke \
  -H "Content-Type: application/json" \
  -d '{"input": {"query": "Hello, how are you?"}}'
```

Response:
```json
{
  "output": "Hello! I'm doing well, thank you...",
  "metadata": {
    "tokensUsed": 42,
    "toolCalls": [],
    "duration": 823
  }
}
```

## Core Concepts

### Agents

Agents are AI-powered units that use LLMs to process input and optionally call tools. Each agent is defined in a YAML file.

**Agent Schema:**
```yaml
name: string                    # Unique identifier (required)
description: string             # Human-readable description (required)
version: string                 # Semantic version (default: "1.0.0")

llm: string | object            # LLM configuration reference
  # Simple: llm: default
  # With override: llm: { name: default, temperature: 0.3 }

prompt:                         # Prompt configuration (required)
  system: string                # System message/instructions
  inputVariables: [string]      # Variables to interpolate in prompt

tools:                          # Available tools (optional)
  - mcp:<server-name>           # MCP server tools
  - knowledge:<store-name>      # Knowledge store search
  - function:<function-name>    # Custom functions
  - builtin:<tool-name>         # Built-in tools (ask_user, save_memory)
  - sandbox:<tool-name>         # Sandboxed execution tools
  - workspace:<tool-name>       # Workspace file tools

skills:                         # Skills to attach (optional)
  - skill-name                  # By name (loads skills/<name>/SKILL.md)
  # Or: { mode: all }           # Attach all available skills

memory: boolean | object        # Persistent agent memory (optional)
  # Simple: memory: true
  # With config: memory: { enabled: true, maxLines: 100 }

publish: boolean | object       # Standalone chat page (optional)
  # Simple: publish: true
  # With password: publish: { enabled: true, password: "secret" }

integrations:                   # External integrations (optional)
  - type: email                 # Email via IMAP/SMTP
    config: { ... }
  - type: collabnook            # WebSocket chat
    config: { ... }

triggers:                       # Automated execution (optional)
  - type: cron
    schedule: "0 9 * * *"       # Cron expression
    input: { ... }
  - type: webhook
    path: "/hook/my-agent"

maxIterations: number           # Override default 200 iteration limit (optional)

sampleQuestions:                # Example prompts shown in UI (optional)
  - "What can you help me with?"
  - "Analyze this data"

output:                         # Output formatting (optional)
  format: text | json | structured
  schema:                       # Required when format is "structured"
    type: object
    properties:
      <field>:
        type: string | number | boolean | array | object
        description: string
        enum: [values]
    required: [field1, field2]
```

**Example Agent with Tools:**
```yaml
name: researcher
description: Researches topics using web fetch and knowledge search
version: "1.0.0"

llm:
  name: default
  temperature: 0.5

prompt:
  system: |
    You are a thorough researcher.
    Use available tools to gather information before responding.
  inputVariables:
    - topic
    - context

tools:
  - mcp:fetch              # Web fetch MCP server
  - knowledge:docs         # Knowledge store search
  - function:custom-tool   # Custom function
  - sandbox:exec           # Sandboxed code execution
  - workspace:read         # Read workspace files

skills:
  - web-pilot

memory: true

publish:
  enabled: true
  password: "research-pass"

sampleQuestions:
  - "Research the latest trends in AI"
  - "What does the knowledge base say about X?"

output:
  format: text
```

**Agent with Structured Output:**
```yaml
name: sentiment-analyzer
description: Analyzes sentiment with structured output
version: "1.0.0"

llm:
  name: default
  temperature: 0

prompt:
  system: |
    Analyze the sentiment of the provided text.
  inputVariables:
    - text

output:
  format: structured
  schema:
    type: object
    properties:
      sentiment:
        type: string
        enum: [positive, negative, neutral]
      confidence:
        type: number
        minimum: 0
        maximum: 1
      keywords:
        type: array
        items:
          type: string
    required:
      - sentiment
      - confidence
```

### Published Agents

Agents with `publish: true` (or `publish: { enabled: true, password: "..." }`) get standalone chat pages at `/chat/<agent-name>`. These pages are independent of the global `AUTH_PASSWORD` and support per-agent password protection.

Features:
- Standalone chat UI with markdown rendering, code highlighting, thinking blocks, tool call visualization
- File/image attachments via drag-and-drop or paste
- Streaming responses with real-time stats
- Per-agent token auth (independent of global auth)
- Accessible at `http://localhost:3000/chat/<agent-name>`

### Skills System

Skills are prompt augmentation files that inject additional context and instructions into agent system prompts at runtime.

**Skill File Format:** `skills/<name>/SKILL.md`
```markdown
---
description: What this skill does
tags: [category1, category2]
sandbox: true  # Optional: indicates skill needs sandbox tools
---

# Skill Instructions

Your skill content here. This markdown is injected into the agent's system prompt.
```

**Attaching Skills to Agents:**
```yaml
# Specific skills
skills:
  - web-pilot
  - pii-guard

# All available skills
skills:
  mode: all
```

### Persistent Memory

Agents can persist memory across sessions to disk. Memory is saved to `.memory/<agent-name>.md` and loaded into the system prompt on each invocation.

```yaml
# Simple
memory: true

# With configuration
memory:
  enabled: true
  maxLines: 100    # Max lines to keep in memory file
```

When memory is enabled, agents receive a `save_memory` built-in tool that allows them to persist important information between conversations.

### Conversation Memory

Agents automatically support conversation memory when a `sessionId` is provided in API calls. No agent-level configuration is required.

```bash
# First message in a session
curl -X POST http://localhost:3000/api/agents/chatbot/invoke \
  -H "Content-Type: application/json" \
  -d '{"input": {"message": "My name is Alice"}, "sessionId": "user-123"}'

# Follow-up message (agent remembers context)
curl -X POST http://localhost:3000/api/agents/chatbot/invoke \
  -H "Content-Type: application/json" \
  -d '{"input": {"message": "What is my name?"}, "sessionId": "user-123"}'
```

Memory settings (global, configured in orchestrator):
- `maxMessagesPerSession`: 50 (default) - FIFO eviction when exceeded
- `sessionTTL`: optional, in milliseconds - automatic cleanup of expired sessions

Session management endpoints:
- `GET /api/agents/sessions/stats` - Session statistics
- `GET /api/agents/sessions/:sessionId` - Get session details
- `DELETE /api/agents/sessions/:sessionId` - Clear session messages

### Workflows

Workflows orchestrate multiple agents. Two types are supported: step-based (`type: steps`) and ReAct (`type: react`).

#### Step-Based Workflows

Step-based workflows define explicit sequences with support for parallel execution, conditional logic, and state management.

**Workflow Schema:**
```yaml
name: string                    # Unique identifier (required)
description: string             # Human-readable description (required)
version: string                 # Semantic version (default: "1.0.0")
type: steps                     # Optional, "steps" is default

chatOutputFormat: text | json   # Controls UI rendering (optional, default: text)
sampleQuestions:                # Example prompts shown in UI (optional)
  - "Run analysis on X"

input:                          # Input schema (required)
  schema:
    <field_name>:
      type: string | number | boolean | array | object
      required: boolean
      default: any
      description: string

steps:                          # Workflow steps (required)
  - id: string                  # Unique step identifier
    agent: string               # Agent name to execute
    input:                      # Input mapping using templates
      <key>: "{{input.field}}"           # From workflow input
      <key>: "{{steps.stepId.output}}"   # From previous step
    condition: string           # Optional conditional execution
    retry:                      # Optional retry configuration
      maxAttempts: number       # Default: 3
      delay: number             # Milliseconds (default: 1000)
    output:
      key: string

  # Parallel execution
  - parallel:
      - id: step1
        agent: agent1
        input: {...}
      - id: step2
        agent: agent2
        input: {...}

config:                         # Workflow configuration (optional)
  timeout: number               # Total timeout ms (default: 300000)
  onError: stop | continue | retry

output:                         # Output mapping (required)
  <key>: "{{steps.stepId.output}}"
```

**Template Syntax:**
- `{{input.fieldName}}` - Access workflow input field
- `{{steps.stepId.output}}` - Access step output
- `{{steps.stepId.output.nested.path}}` - Access nested output
- `{{steps.stepId.metadata.duration}}` - Access step metadata

**Example Step-Based Workflow:**
```yaml
name: research-paper
description: Research a topic and write a comprehensive paper
version: "1.0.0"

sampleQuestions:
  - "Write a paper about quantum computing"

input:
  schema:
    topic:
      type: string
      required: true
      description: The topic to research
    style:
      type: string
      default: "professional"

steps:
  - id: research
    agent: researcher
    input:
      topic: "{{input.topic}}"
      context: "Gather comprehensive information"
    output:
      key: researchFindings

  - id: summarize
    agent: summarizer
    input:
      content: "{{steps.research.output}}"
      maxPoints: "10"
    condition: "{{steps.research.metadata.success}}"
    output:
      key: summary

  - id: write
    agent: writer
    input:
      research: "{{steps.research.output}}"
      outline: "{{steps.summarize.output}}"
      style: "{{input.style}}"
    output:
      key: paper

config:
  timeout: 600000
  onError: stop

output:
  paper: "{{steps.write.output}}"
  summary: "{{steps.summarize.output}}"
  researchFindings: "{{steps.research.output}}"
```

#### ReAct Workflows

Autonomous, prompt-driven workflows using the ReAct pattern. The agent decides which tools and agents to call based on the prompt.

**ReAct Schema:**
```yaml
name: string                    # Unique identifier (required)
description: string             # Human-readable description (required)
version: string                 # Semantic version (default: "1.0.0")
type: react                     # Required for ReAct workflows

chatOutputFormat: text | json   # Controls UI rendering (optional, default: text)
sampleQuestions:                # Example prompts shown in UI (optional)
  - "Research topic X"

input:                          # Input schema (required)
  schema:
    <field_name>:
      type: string | number | boolean | array | object
      required: boolean
      description: string

prompt:                         # Prompt configuration (required)
  system: string                # System message with instructions
  goal: string                  # Goal template (supports {{input.*}} interpolation)

graph:                          # ReAct configuration (required)
  model: string                 # LLM config name from llm.json

  executionMode: react | single-turn  # Default: react
  # react: Full ReAct loop, multiple rounds of tool calls
  # single-turn: Calls tools once and returns

  tools:                        # Tool discovery config
    mode: all | include | exclude | none  # Default: all
    sources:                    # Tool source types (default: all)
      - mcp
      - knowledge
      - function
      - builtin
    include: [string]           # Tool names to include (for mode: include)
    exclude: [string]           # Tool names to exclude (for mode: exclude)

  agents:                       # Agent discovery config
    mode: all | include | exclude | none  # Default: all
    include: [string]           # Agent names to include
    exclude: [string]           # Agent names to exclude

  maxIterations: number         # Max tool-calling iterations (default: 10)
  timeout: number               # Timeout in ms (default: 300000)

output:                         # Output extraction (required)
  <key>: "{{state.messages[-1].content}}"

config:                         # Workflow configuration (optional)
  onError: stop | continue | retry
```

**Execution Modes:**

| Mode | Behavior | Best For |
|------|----------|----------|
| `single-turn` | Calls tools once, then returns | Research, data gathering, straightforward tasks |
| `react` | Multiple rounds of tool calls with reasoning | Complex multi-step problems, iterative analysis |

**Human-in-the-Loop:** ReAct workflows support the `builtin:ask_user` tool. When called, the workflow pauses and waits for user input. Resume with `POST /api/workflows/:name/resume`. Include `builtin` in tool sources to enable it.

**Example ReAct Workflow:**
```yaml
name: react-research
description: Autonomous research using tool discovery
version: "1.0.0"
type: react

sampleQuestions:
  - "Research the latest AI developments"

input:
  schema:
    topic:
      type: string
      required: true

prompt:
  system: |
    You are a research assistant with access to tools and agents.
    Identify all tools you need, call them in parallel, then synthesize results.
  goal: "Research and analyze: {{input.topic}}"

graph:
  model: default
  executionMode: single-turn
  tools:
    mode: all
    sources: [mcp, knowledge, function, builtin]
  agents:
    mode: all
  maxIterations: 10
  timeout: 300000

output:
  analysis: "{{state.messages[-1].content}}"
```

### Knowledge Stores

Knowledge stores enable semantic search and RAG capabilities. All stores use **SQLite with sqlite-vec** as the unified persistence layer — no external vector databases required. They are configured in `*.knowledge.yaml` files in the `knowledge/` directory. Optionally add a `graph.directMapping` section to build a knowledge graph from structured data.

**Knowledge Store Schema:**
```yaml
name: string                    # Unique identifier (required)
description: string             # Human-readable description (required)

source:                         # Data source (required)
  # Directory source
  type: directory
  path: string                  # Path relative to project root
  pattern: string               # Glob pattern for directories
  recursive: boolean            # Recursive search (default: true)

  # File source
  type: file
  path: string                  # Single file path

  # Database source
  type: database
  connectionString: string      # postgresql:// or mysql://
  query: string                 # SQL query
  contentColumn: string         # Column with content (default: "content")
  metadataColumns: [string]     # Columns for metadata
  batchSize: number             # Rows per batch (default: 100)

  # Web source
  type: web
  url: string                   # URL to scrape
  selector: string              # CSS selector (optional)
  headers:                      # Custom headers (optional)
    Authorization: "Bearer TOKEN"

loader:                         # Document loader (optional)
  type: text | pdf | csv | json | markdown | html
  # Defaults: html for web sources, text for file/directory sources
  # Not used for database sources

  jsonPath: string              # Dot-notation path to extract nested array from JSON (optional)
                                # e.g., "items" or "data.results" — web sources only

splitter:                       # Text chunking (required)
  type: character | recursive | token | markdown
  chunkSize: number             # Characters per chunk (default: 1000)
  chunkOverlap: number          # Overlap between chunks (default: 200)

embedding: string               # Reference to embedding config (default: "default")

metadata:                       # Custom metadata (optional)
  key: value

reindex:                        # Automatic reindexing (optional)
  schedule: string              # Cron expression, e.g., "0 */6 * * *" for every 6 hours

graph:                          # Optional — enables entity graph
  directMapping:                # Maps structured data to entities and relationships
    entities:
      - type: string            # Entity type name
        idColumn: string        # Column used as unique ID
        nameColumn: string      # Column used as display name (optional)
        properties: [string]    # Columns to include as entity properties
    relationships:              # Optional
      - type: string            # Relationship type name
        source: string          # Source entity type
        target: string          # Target entity type
        sourceIdColumn: string  # Column linking to source entity
        targetIdColumn: string  # Column linking to target entity

search:                         # Search configuration (optional)
  defaultK: number              # Results per search (default: 4)
  scoreThreshold: number        # Minimum similarity (0-1)
```

**How it works:**
- All data persists to SQLite at `.knowledge-data/{name}.db`
- On restart, source hashes are compared — if unchanged, data restores instantly without re-indexing
- Stores with `graph.directMapping` also store entities and relationships with vector embeddings
- Agents get additional graph tools (traverse, entity_lookup, graph_schema) when entities exist
- Search combines chunk KNN + entity KNN + neighborhood expansion, merged by score
- Stores with `reindex.schedule` automatically refresh on a schedule via node-cron

**Example Vector-Only Store:**
```yaml
name: docs
description: Knowledge base for semantic search

source:
  type: directory
  path: knowledge/sample-data
  pattern: "*.txt"

loader:
  type: text

splitter:
  type: character
  chunkSize: 1000
  chunkOverlap: 200

embedding: default

search:
  defaultK: 4
  scoreThreshold: 0.2
```

**Example PDF Knowledge Store:**
```yaml
name: patient-records
description: Deidentified patient records

source:
  type: directory
  path: knowledge/pdfs
  pattern: "*.pdf"

loader:
  type: pdf

splitter:
  type: recursive
  chunkSize: 2000
  chunkOverlap: 200

embedding: default
```

**Example Web Source with JSON Path:**
```yaml
name: api-data
description: Data from a REST API

source:
  type: web
  url: https://api.example.com/data
  headers:
    Authorization: "Bearer ${API_TOKEN}"

loader:
  type: json
  jsonPath: "data.results"

splitter:
  type: recursive
  chunkSize: 1000
  chunkOverlap: 100

embedding: default

reindex:
  schedule: "0 */6 * * *"
```

**Example Store with Graph (Direct Mapping):**
```yaml
name: call-center-analysis
description: Call center data with agent/customer knowledge graph

source:
  type: database
  connectionString: postgresql://user:pass@localhost:5432/calls
  query: |
    SELECT c.id, c.transcript AS content, c.outcome,
           a.name AS agent_name, a.id AS agent_id,
           cu.name AS customer_name, cu.id AS customer_id
    FROM calls c
    JOIN agents a ON c.agent_id = a.id
    JOIN customers cu ON c.customer_id = cu.id
  contentColumn: content
  metadataColumns: [id, outcome, agent_name, agent_id, customer_name, customer_id]

loader:
  type: text

splitter:
  type: recursive
  chunkSize: 2000
  chunkOverlap: 200

embedding: default

graph:
  directMapping:
    entities:
      - type: Agent
        idColumn: agent_id
        nameColumn: agent_name
        properties: [agent_name]
      - type: Customer
        idColumn: customer_id
        nameColumn: customer_name
        properties: [customer_name]
      - type: Call
        idColumn: id
        nameColumn: id
        properties: [outcome]
    relationships:
      - type: HANDLED_BY
        source: Call
        target: Agent
        sourceIdColumn: id
        targetIdColumn: agent_id
      - type: MADE_BY
        source: Call
        target: Customer
        sourceIdColumn: id
        targetIdColumn: customer_id

search:
  defaultK: 10
```

### Functions

Functions are custom JavaScript tools that extend agent capabilities. They require no dependencies and are simple to create.

**Function Schema:**
```javascript
export default {
  name: 'function-name',           // Unique identifier (required)
  description: 'What it does',     // Clear description (required)

  parameters: {                    // Input parameters (required)
    param1: {
      type: 'number',              // string | number | boolean | array | object | enum
      description: 'Parameter description',
      required: true,              // Optional, defaults to true
      default: 0,                  // Optional default value
    },
  },

  execute: async ({ param1 }) => { // Execution function (required)
    // Your logic here
    return `Result: ${param1}`;
  },
};

// Optional metadata
export const metadata = {
  name: 'function-name',
  version: '1.0.0',
  author: 'Your Name',
  tags: ['category'],
};
```

**Example Function (Calculator):**
```javascript
export default {
  name: 'calculator',
  description: 'Performs basic arithmetic operations',

  parameters: {
    a: { type: 'number', description: 'First number' },
    b: { type: 'number', description: 'Second number' },
    operation: {
      type: 'enum',
      values: ['add', 'subtract', 'multiply', 'divide'],
      description: 'Operation to perform',
    },
  },

  execute: async ({ a, b, operation }) => {
    switch (operation) {
      case 'add': return `${a} + ${b} = ${a + b}`;
      case 'subtract': return `${a} - ${b} = ${a - b}`;
      case 'multiply': return `${a} * ${b} = ${a * b}`;
      case 'divide': return `${a} / ${b} = ${a / b}`;
    }
  },
};
```

**Using Functions in Agents:**
```yaml
name: math-assistant
description: Assistant that can perform calculations

llm:
  name: default
  temperature: 0.3

prompt:
  system: |
    You are a math assistant.
    Use the calculator tool to perform calculations.
  inputVariables:
    - query

  - function:calculator    # References calculator.function.js

  format: text
```

### MCP Servers

Model Context Protocol (MCP) servers provide external tools to agents. Configure them in mcp.json.

**MCP Configuration:**
```json
{
  "version": "1.0.0",
  "servers": {
    "<server-name>": {
      "transport": "streamable-http | stdio | sse | sse-only",
      "url": "https://server-url/mcp",
      "headers": { "key": "value" },
      "command": "node",
      "args": ["./mcp-server.js"],
      "env": { "KEY": "VALUE" },
      "timeout": 30000,
      "enabled": true,
      "description": "Server description"
    }
  },
  "globalOptions": {
    "throwOnLoadError": false,
    "prefixToolNameWithServerName": true,
    "additionalToolNamePrefix": "",
    "defaultToolTimeout": 30000
  }
}
```

**Transport Types:**

| Transport | Use Case | Required Fields |
|-----------|----------|----------------|
| `stdio` | Local CLI tools | `command`, `args` |
| `streamable-http` | Remote HTTP servers | `url` |
| `sse` | Server-Sent Events | `url` |
| `sse-only` | SSE without HTTP fallback | `url` |

**Example MCP Configuration:**
```json
{
  "version": "1.0.0",
  "servers": {
    "fetch": {
      "transport": "streamable-http",
      "url": "https://remote.mcpservers.org/fetch/mcp",
      "description": "Web fetch capabilities",
      "timeout": 30000,
      "enabled": true
    },
    "filesystem": {
      "transport": "stdio",
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/tmp"],
      "description": "File system access"
    }
  }
}
```

**Using MCP Tools in Agents:**
```yaml
tools:
  - mcp:fetch    # All tools from "fetch" server
```

### Agent Orcha Studio

Agent Orcha Studio is a built-in web dashboard available at `http://localhost:3000` when the server is running. It provides a visual interface for managing and testing your entire Agent Orcha setup. When `AUTH_PASSWORD` is set, the Studio will prompt for the password before granting access.

**Studio Tabs:**

| Tab | Description |
|-----|-------------|
| **Agents** | Browse agents, invoke with custom input, stream responses, manage sessions, workflow chat integration (start, stream, interrupt/resume) |
| **Knowledge** | Browse and search knowledge stores, view entities and graph structure for stores with direct mapping |
| **MCP** | Browse MCP servers, view available tools, call tools directly |
| **Skills** | Browse and inspect available skills |
| **Local LLM** | Engine management with tabs for llama-cpp, mlx-serve, Ollama, and LM Studio. Model download/activation, context size config, API key management, embedding config |

### Integrations

Integrations connect agents to external platforms. Configured in agent YAML via the `integrations` field.

**Email Integration (IMAP + SMTP):**
```yaml
integrations:
  - type: email
    config:
      imap:
        host: imap.gmail.com
        port: 993
        user: ${EMAIL_USER}
        password: ${EMAIL_PASSWORD}
      smtp:
        host: smtp.gmail.com
        port: 587
        user: ${EMAIL_USER}
        password: ${EMAIL_PASSWORD}
      fromName: "My Agent"
      fromAddress: ${EMAIL_USER}
      pollInterval: 60000       # Check every 60 seconds
      folder: INBOX
```

When an agent has email integration, it receives inbound emails as triggers and gets an `email_send` tool for composing new emails.

**Collabnook Integration (WebSocket Chat):**
```yaml
integrations:
  - type: collabnook
    config:
      url: ws://localhost:8080
      channel: general
```

Agents receive messages via @mention and reply in-channel.

**Auto-injected Tools:**
- `integration_post` — Post messages to connected channels
- `integration_context` — Get recent channel context
- `email_send` — Send emails (agents with email integration)

### Triggers

Triggers automate agent execution on schedules or external events.

**Cron Trigger:**

triggers:
  - type: cron
    schedule: "0 9 * * *"       # Every day at 9am

**Webhook Trigger:**
```yaml
triggers:
  - type: webhook
    path: "/hook/my-agent"
```

Webhook triggers create an endpoint at `POST /api/triggers/webhook/<path>`. The request body is passed as agent input.

### Engine Registry

The engine registry provides a pluggable architecture for local inference engines. Each engine implements a standardized interface for model loading, unloading, and status reporting.

**Supported Engines:**

| Engine | Format | Platform | Managed |
|--------|--------|----------|---------|
| `llama-cpp` | GGUF | Cross-platform | Yes (auto binary management) |
| `mlx-serve` | MLX SafeTensors | Apple Silicon | Yes (auto binary management) |
| `ollama` | Ollama models | Cross-platform | No (external) |
| `lmstudio` | Any supported | Cross-platform | No (external) |

Managed engines (llama-cpp, mlx-serve) handle binary downloads, model loading, and process lifecycle automatically. External engines (Ollama, LM Studio) are probed for availability and controlled via their APIs.

**Engine API Endpoints:**
- `GET /api/local-llm/engines` — List all engines with status
- `POST /api/local-llm/engines/activate` — Activate model on an engine
- `POST /api/local-llm/engines/context` — Update context size
- `POST /api/local-llm/engines/unload` — Unload model from engine
- `GET /api/local-llm/status` — Current engine and model status
- `GET /api/local-llm/models` — Available models

### Environment Variable Substitution

All config files (agent YAML, knowledge YAML, workflow YAML, skill frontmatter, `llm.json`, `mcp.json`, `sandbox.json`) support environment variable placeholders.

**Syntax:**
- `${ENV_VAR}` — Replaced with the value of `ENV_VAR` from `process.env`
- `${ENV_VAR:-default}` — Uses `default` if `ENV_VAR` is not set

**Example:**
```yaml
# In agent YAML
integrations:
  - type: email
    config:
      imap:
        password: ${EMAIL_PASSWORD}

# In llm.json
{
  "apiKey": "${OPENAI_API_KEY}"
}
```

### Authentication

When `AUTH_PASSWORD` is set, all `/api/*` routes require a valid session cookie. The `/health` endpoint is always public. When `AUTH_PASSWORD` is not set, authentication is disabled and all routes are open. Login is rate-limited to 5 attempts per 60 seconds.

## Library Usage

Agent Orcha can be used as a TypeScript/JavaScript library in your projects.

```typescript
import { Orchestrator } from 'agent-orcha';

const orchestrator = new Orchestrator({
  workspaceRoot: './my-agents-project'
});

await orchestrator.initialize();

// Invoke an agent
const result = await orchestrator.agents.invoke('researcher', {
  topic: 'machine learning',
  context: 'brief overview'
});

console.log(result.output);

// Invoke an agent with session memory
const chatResult = await orchestrator.agents.invoke(
  'chatbot',
  { message: 'Hello' },
  { sessionId: 'user-123' }
);

// Run a workflow
const workflowResult = await orchestrator.workflows.run('research-paper', {
  topic: 'artificial intelligence'
});

console.log(workflowResult.output);

// Search knowledge store
const searchResults = await orchestrator.knowledge.search('docs', {
  query: 'semantic search',
  k: 4
});

console.log(searchResults);

// Clean up
await orchestrator.close();
```

## Tool Types Reference

1. **MCP Tools** - External tools from MCP servers
   ```yaml
   tools:
     - mcp:fetch              # All tools from "fetch" server
     - mcp:filesystem         # All tools from "filesystem" server
   ```

2. **Knowledge Tools** - Semantic search and graph tools on knowledge stores
   ```yaml
   tools:
     - knowledge:docs         # Search "docs" knowledge store
   ```
   When a knowledge store has entities, agents also get these tools:
   - `<store>_search` — Semantic search across chunks and entities
   - `<store>_traverse` — Walk the entity graph
   - `<store>_entity_lookup` — Look up specific entities
   - `<store>_graph_schema` — Get the graph schema (entity types, relationship types)
   - `<store>_sql` — Run read-only SQL against the store's SQLite database

3. **Function Tools** - Custom JavaScript/TypeScript functions in functions/
   ```yaml
   tools:
     - function:calculator
     - function:fibonacci
   ```

4. **Built-in Tools** - Framework-provided tools
   ```yaml
   tools:
     - builtin:ask_user       # Human-in-the-loop (pauses for user input)
   ```
   Auto-injected built-in tools (no explicit declaration needed):
   - `save_memory` — Save to persistent memory (when `memory: true`)
   - `integration_post` — Post to connected channels (when integrations configured)
   - `integration_context` — Get channel context (when integrations configured)
   - `email_send` — Send emails (when email integration configured)

5. **Sandbox Tools** - Sandboxed execution in Docker container
   ```yaml
   tools:
     # VM execution
     - sandbox:exec           # Execute code in sandbox VM
     - sandbox:web_fetch      # Fetch URLs from sandbox (with SSRF protection)
     - sandbox:web_search     # Web search from sandbox

     # Shell
     - sandbox:shell          # Execute shell commands as non-root sandbox user

     # DOM-based browser tools
     - sandbox:browser_navigate    # Navigate to URL
     - sandbox:browser_observe     # Get page structure and text excerpt
     - sandbox:browser_click       # Click elements by CSS selector
     - sandbox:browser_type        # Type text into form fields
     - sandbox:browser_screenshot  # Take screenshot (returns image)
     - sandbox:browser_evaluate    # Run JavaScript in page context

     # Vision browser tools (EXPERIMENTAL_VISION=true)
     - sandbox:vision_screenshot   # Take screenshot for vision analysis
     - sandbox:vision_navigate     # Navigate to URL
     - sandbox:vision_click        # Click at pixel coordinates
     - sandbox:vision_type         # Type text at current focus
     - sandbox:vision_scroll       # Scroll page
     - sandbox:vision_key          # Press keyboard key
     - sandbox:vision_drag         # Drag between coordinates

     # Sandboxed file tools
     - sandbox:file_read           # Read files (scoped to /tmp)
     - sandbox:file_write          # Write files (scoped to /tmp)
     - sandbox:file_edit           # Edit files (scoped to /tmp)
     - sandbox:file_insert         # Insert lines (scoped to /tmp)
     - sandbox:file_replace_lines  # Replace line ranges (scoped to /tmp)
   ```

6. **Workspace Tools** - Workspace file access
   ```yaml
   tools:
     - workspace:read              # Read workspace files
     - workspace:write             # Write workspace files
     - workspace:delete            # Delete files/directories
     - workspace:list              # List directory contents
     - workspace:list_resources    # List all loaded resources
     - workspace:diagnostics       # Get workspace diagnostics
   ```

### Configuration Best Practices

1. **LLM Configuration**: Store API keys in environment variables, reference via `${ENV_VAR}` in llm.json
2. **Knowledge Stores**: All stores use SQLite — data persists automatically to `.knowledge-data/`
3. **Timeouts**: Configure appropriate timeouts for workflows based on expected duration
4. **Error Handling**: Set onError strategy in workflows (stop, continue, retry)
5. **CORS**: Default is same-origin only; set `CORS_ORIGIN` for cross-origin access
6. **Auth**: Always set `AUTH_PASSWORD` in production

### Example 6: Knowledge Store with Graph (Direct Mapping)

```yaml
# knowledge/org-chart.knowledge.yaml
name: org-chart
description: Organization chart with employee/department graph

source:
  type: database
  connectionString: postgresql://user:pass@localhost:5432/hr
  query: |
    SELECT e.id, e.name, e.title, e.bio AS content,
           d.name AS dept_name, d.id AS dept_id
    FROM employees e
    JOIN departments d ON e.department_id = d.id
  contentColumn: content
  metadataColumns: [id, name, title, dept_name, dept_id]

loader:
  type: text

splitter:
  type: recursive
  chunkSize: 2000
  chunkOverlap: 200

embedding: default

graph:
  directMapping:
    entities:
      - type: Employee
        idColumn: id
        nameColumn: name
        properties: [name, title]
      - type: Department
        idColumn: dept_id
        nameColumn: dept_name
        properties: [dept_name]
    relationships:
      - type: WORKS_IN
        source: Employee
        target: Department
        sourceIdColumn: id
        targetIdColumn: dept_id

search:
  defaultK: 10
```

## Troubleshooting

### Common Issues

1. **LLM Connection Errors**
   - Verify API keys in llm.json or environment variables
   - Check baseUrl for local models (LM Studio, Ollama) — configure via engineUrls in llm.json
   - Ensure model name matches provider configuration
   - For local engines, check `GET /api/local-llm/engines` for status

2. **Knowledge Store Errors**
   - Check embedding model configuration in llm.json
   - Ensure source documents exist at specified paths
   - Verify SQLite write permissions for `.knowledge-data/` directory
   - For PDF stores, ensure `pdf-parse` dependency is available

3. **MCP Server Errors**
   - Verify MCP server is accessible at specified URL
   - Check timeout configuration
   - Review MCP server logs for errors

4. **Workflow Execution Errors**
   - Check step dependencies and template syntax
   - Verify agent names referenced in steps exist
   - Review timeout configuration for long-running workflows
   - For ReAct: Check maxIterations to prevent runaway loops
   - Use `POST /api/workflows/:name/resume` for interrupted human-in-the-loop workflows

5. **Session/Memory Issues**
   - Ensure consistent sessionId across requests
   - Check session TTL hasn't expired
   - Session memory is in-process; sessions are lost on server restart
   - Persistent memory (`memory: true`) survives restarts via `.memory/` directory

6. **Sandbox/Browser Issues**
   - Ensure Docker container has `BROWSER_SANDBOX=true` set
   - Check CDP connectivity at port 9222
   - Vision browser requires `EXPERIMENTAL_VISION=true`
